## 1. Problem

A user types *"build me a weekly portfolio check-in"* into ainative chat today. Despite having:

- 13 shipped agent profiles (`wealth-manager`, `researcher`, `code-reviewer`, ...)
- 10 shipped workflow blueprints (`investment-research`, `business-daily-briefing`, ...)
- Shipped composition tools (`create_profile`, `create_blueprint`, `create_schedule`, `create_table`)
- A shipped `AppMaterializedCard` + `detectComposedApp` pipeline (Phase 2+3)
- A shipped `create_plugin_spec` chat tool + `ExtensionFallbackCard` component (Phase 6)

...the user gets a plain-text conversational reply. None of the beautiful composition infrastructure fires. The LLM isn't told *when* to use those primitives, and the `ExtensionFallbackCard` is never rendered.

M4.5 closes the loop: classify inbound chat intent, nudge the LLM toward the right tool calls when the ask is an "app" request, and surface the plugin-scaffold fallback when composition can't express the ask.

## 2. Goals

- A user can type a natural-language app request and watch `AppMaterializedCard` fire with the right primitives composed, without any slash-command or structured input.
- A user can type a plugin-shaped request ("I need a tool that calls the GitHub API") and see `ExtensionFallbackCard` inline with pre-inferred scaffold inputs, click "Scaffold + open", and land on a working `.ainative/plugins/<id>/` directory.
- Non-app conversations pass through untouched — no regressions in existing chat behavior.
- The classifier is deterministic, testable, and cheap (no new LLM hop).
- Accuracy tunable via a curated trigger phrase list; confidence is binary (match/no-match) in v1.

## 3. Non-goals (deferred to M4.6+)

- LLM-based intent classification (would warrant a new TDR).
- Multi-turn app understanding ("also make it track ratings").
- Numeric confidence scoring / user-facing dry-run mode.
- Dynamic primitive-map updates from registry at runtime (v1 keys the map statically).
- Auto-invocation of `create_plugin_spec` without user click — the scaffold card stays explicit.
- Planner analytics, "was this helpful?" chips, A/B-testable trigger variants.
- Cross-runtime parity — v1 wires into the Claude SDK `sendMessage` path only. Codex / Ollama engines keep the existing behavior.

## 4. Architecture

### 4.1 Hook point

One site in `src/lib/chat/engine.ts`: **before** the `query()` call, after `buildChatContext()` returns. The planner receives:

- the user's current message string,
- the just-built `ChatContext` (for project-id + history access if needed later — v1 uses current message only),
- the available profile + blueprint inventory (imported from existing registries).

It returns a `ClassifierVerdict` discriminated union (§4.3). The engine branches on `verdict.kind` before dispatching.

### 4.3 Contract — `ClassifierVerdict`

```ts
// src/lib/chat/planner/types.ts

import type { CreatePluginSpecInput } from "@/lib/chat/tools/plugin-spec-tools";

export interface ComposePlan {
  /** Profile id from the builtin registry (e.g., "wealth-manager"). */
  profileId: string;
  /** Blueprint id from the builtin registry (e.g., "investment-research"). */
  blueprintId: string;
  /** Optional tables to create alongside the profile+blueprint. */
  tables?: Array<{
    name: string;
    columns: Array<{ name: string; type: "text" | "number" | "boolean" | "date" }>;
  }>;
  /** Optional schedule to create (cron string). */
  schedule?: { cron: string; description: string };
  /** Human-readable rationale for the telemetry + system-prompt hint. */
  rationale: string;
}

export interface ScaffoldPlan {
  /** Proposed inputs for create_plugin_spec. */
  plugin: CreatePluginSpecInput;
  /** Human-readable rationale surfaced in the assistant preamble. */
  rationale: string;
}

export type ClassifierVerdict =
  | { kind: "compose"; plan: ComposePlan }
  | { kind: "scaffold"; plan: ScaffoldPlan }
  | { kind: "conversation" };

export interface PlannerContext {
  /** Project id for scoping, if any. */
  projectId: string | null;
  /** Prior chat turns (v1 unused, reserved for v2). */
  history: Array<{ role: "user" | "assistant"; content: string }>;
}
```

**Why a discriminated union:** callers pattern-match on `kind`, TypeScript enforces shape. Adding a fourth verdict later (e.g., `query`) is a single case addition.

### 4.4 Classifier algorithm (v1 — pattern-based)

```ts
// classifier.ts — pseudo-code

export function classifyMessage(
  message: string,
  ctx: PlannerContext
): ClassifierVerdict {
  const normalized = message.toLowerCase().trim();

  // Step 1: scaffold signals (checked first — more specific)
  for (const phrase of SCAFFOLD_TRIGGERS) {
    if (normalized.includes(phrase)) {
      const plan = inferScaffoldPlan(message);
      if (plan) return { kind: "scaffold", plan };
    }
  }

  // Step 2: compose signals
  for (const phrase of COMPOSE_TRIGGERS) {
    if (normalized.includes(phrase)) {
      const plan = inferComposePlan(message);
      if (plan) return { kind: "compose", plan };
    }
  }

  // Step 3: default
  return { kind: "conversation" };
}
```

**Order matters:** scaffold-first. A message like *"I need a tool that pulls my GitHub issues so I can build an app to summarize them"* hits both triggers; scaffold-first respects user intent (they want a plugin for integration, not a profile).

**`inferComposePlan(message)` → `ComposePlan | null`:**

1. Extract domain keywords via a substring scan against `primitive-map.ts` keys.
2. Pick the highest-scoring (longest-prefix) match.
3. Map the match to `{ profileId, blueprintId, tables?, schedule? }`.
4. Generate a rationale string like: `"Matched 'portfolio' → wealth-manager + investment-research"`.
5. Return null if no keyword match (fall through to conversation).

**`inferScaffoldPlan(message)` → `ScaffoldPlan | null`:**

1. Extract a plausible plugin id via noun-phrase heuristics (`/(?:github|jira|linear|slack|...)/` + post-processing).
2. Derive language defaults: `"python"` (v1 only supports python bodies per Phase 6).
3. Derive transport defaults: `"stdio"` (v1 only).
4. Best-effort tool-name inference from verbs in the message (`"pull" → "fetch_issues"`, etc.). If none inferable, default to a single `"run"` tool.
5. Return null if no plugin-name-y noun found — the message is probably just venting.

**Why heuristic pattern matching:** no new LLM hop; deterministic; testable; cheap. Accuracy ceiling is acceptable for v1 because the worst failure mode is a silent fall-through to conversation, which is how chat already behaves. **Does not warrant TDR** (reuses existing registries + chat-tool pathways).

### 4.5 Compose path — prompt augmentation

When verdict is `{ kind: "compose", plan }`, the engine builds a `composition-hint.ts`:buildCompositionHint(plan)` block and appends it to `fullPrompt` before the user turn:

```
## App Composition Hint (M4.5 planner, 2026-04-21)

The user's message appears to be an app-creation request.
Recommended composition:

- Profile: `wealth-manager` (existing builtin — list_profiles to confirm)
- Blueprint: `investment-research` (existing builtin — list_blueprints to confirm)
- Tables: create a `positions` table with columns: ticker (text), shares (number), cost_basis (number)

Rationale: Matched 'portfolio' → wealth-manager + investment-research.

Your next actions should be:
1. Call `create_profile` with the profile id and an app-namespaced artifact id.
2. Call `create_blueprint` with the blueprint id and a namespaced artifact id.
3. Call `create_table` for each proposed table.
4. Respond to the user summarizing what was composed.

If the user's actual need differs from this hint (e.g., they want a different
profile, or no tables), prefer their stated intent over this hint.
```

**Key properties:**

- Hint is **advisory**, not prescriptive — LLM can override based on user intent.
- Hint references existing primitives (profile/blueprint ids) that the LLM already knows about via the existing system prompt's tool catalog.
- The existing `detectComposedApp(toolResults)` in `engine.ts:708` fires after the LLM dispatches the tool calls — no new detection work needed.

### 4.6 Scaffold path — short-circuit + metadata

When verdict is `{ kind: "scaffold", plan }`, the engine **skips the `query()` call** and instead:

1. Streams a short canned preamble delta: `"I can scaffold a plugin for that. Here's what I'd generate:"` (with the rationale appended).
2. Persists the assistant message with metadata:
   ```json
   {
     "modelId": "<target model>",
     "runtimeId": "<target runtime>",
     "extensionFallback": {
       "plugin": { "id": "github-mine", "language": "python", "transport": "stdio", "tools": [...] },
       "rationale": "Matched 'pull my GitHub' → plugin fallback (external API integration)."
     }
   }
   ```
3. Emits `{ type: "done", messageId }`.
4. `chat-message.tsx` reads `metadata.extensionFallback` and renders `<ExtensionFallbackCard initialState="prompt" plugin={meta.extensionFallback.plugin} onScaffold={defaultScaffoldHandler} onTryAlt={defaultTryAltHandler} />`.

**Short-circuit rationale:** the LLM would produce the same preamble text and wouldn't add value. Skipping it saves cost, latency, and one LLM turn. Aligns with brainstorming decision #3 ("canned message is OK").

### 4.7 Card wiring — `ExtensionFallbackCard` defaults

Phase 6 shipped the card as renderable-only. M4.5 adds **default handlers**:

**`defaultScaffoldHandler(input: CreatePluginSpecInput)`:**

```ts
const res = await fetch("/api/plugins/scaffold", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify(input),
});
if (!res.ok) {
  const err = await res.json().catch(() => ({ error: "Unknown error" }));
  throw new Error(err.error ?? `HTTP ${res.status}`);
}
return res.json() as Promise<{ pluginDir: string; files: string[] }>;
```

**`defaultTryAltHandler(message: string)`:**

Sends a new user turn to the same conversation: `"Compose an app without a plugin for: <original message>"`. Re-enters the classifier; typically lands on the compose path.

**API route shape** (`src/app/api/plugins/scaffold/route.ts`):

```ts
export async function POST(req: NextRequest) {
  const input = CreatePluginSpecInputSchema.parse(await req.json());
  try {
    const result = await scaffoldPluginSpec(input);
    return NextResponse.json(result);
  } catch (err) {
    if (err instanceof PluginSpecAlreadyExistsError) {
      return NextResponse.json({ error: err.message, code: "already_exists" }, { status: 409 });
    }
    if (err instanceof PluginSpecInvalidIdError) {
      return NextResponse.json({ error: err.message, code: "invalid_id" }, { status: 400 });
    }
    if (err instanceof PluginSpecWriteError) {
      return NextResponse.json({ error: err.message, code: "write_failed" }, { status: 500 });
    }
    throw err;
  }
}
```

## 7. Error handling

**Classifier errors:** the classifier is pure + total (no throws). Any unexpected input falls through to `{ kind: "conversation" }`. Defense-in-depth: wrap the classifier call in the engine in a try/catch that logs + defaults to conversation on throw. This preserves existing chat behavior in the worst case.

**Scaffold short-circuit errors:**

- If `addMessage()` for the canned preamble fails, the engine falls back to the normal `query()` path. User sees the LLM-generated response instead of the card. Classifier verdict is logged to `recordTermination` telemetry.
- If the metadata persist fails after the preamble was streamed, we log + emit `{ type: "error", message: "Failed to persist assistant message" }`. User sees the preamble but no card; they can re-try.

**API route errors (§4.7):** three named error types from Phase 6 surface as `{ code }` values for future UX branching.

**Card handler errors:** `defaultScaffoldHandler` throws on non-2xx; `ExtensionFallbackCard`'s existing "failed" state renders with the error message.

## 8. Rollout

### 8.1 Not feature-flagged in v1

The planner runs on every chat turn in the Claude SDK engine (`engine.ts`). Rationale:

- Classifier is ~1ms synchronous work (regex + hash lookups).
- Conservative trigger list minimizes false positives; fall-through-to-conversation is the safe default.
- Feature flags add surface area without a testable hypothesis in v1.

If the classifier produces unacceptable false positives in practice, add `AINATIVE_PLANNER_ENABLED=false` opt-out in M4.6 as a kill switch. Not worth anticipating pre-ship.

### 8.2 Codex / Ollama engine parity

v1 wires only the Claude SDK engine. `codex-engine.ts` and `ollama-engine.ts` skip the classifier. Tracked for M4.6. This is acceptable because:

- The signature demo runs on Claude by default.
- Codex + Ollama users can still invoke composition via explicit tool calls — nothing regresses.
- Planner extraction is additive; Codex/Ollama parity is a lift, not a redesign.

### 8.3 Cost + latency envelope

- **Classifier:** ~1ms per turn (regex + Map lookup).
- **Compose path:** zero added latency (prompt is larger by ~400 chars — immeasurable at current token rates).
- **Scaffold path:** **negative latency** — skips an LLM turn.
- **API route:** O(scaffoldPluginSpec), same as direct chat-tool invocation.

No cost amplification. No new external calls.

## 9. Security considerations

**Classifier is pure + client-trusted.** Inputs are chat messages already accepted by the chat pipeline. No new input surface.

**API route `/api/plugins/scaffold`** calls `scaffoldPluginSpec` which writes to `~/.ainative/plugins/<id>/`. Already part of the Phase 6 threat model:

- Reserved id list protects the echo-server fixture.
- Slug regex prevents path traversal.
- `author: "ainative"` + `origin: "ainative-internal"` ensures TDR-037 classifier signals #1 + #2 both fire → self-extension path bypasses the lockfile per `handoff/2026-04-21-tdr-037-phase-6-shipped-handoff.md` §"Regression guards".

**No new auth surface.** The API route inherits the app's existing auth posture (localhost-only dev server, no multi-tenant assumptions).

## 10. Instrumentation

- `recordTermination` entries gain an optional `plannerVerdict` field: `"compose" | "scaffold" | "conversation"`. Enables post-hoc accuracy tuning.
- No new DB tables or columns in v1.
