# Living Book Self-Updating Workflow

## Description

Build a Stagent workflow that automatically regenerates Book chapters when their corresponding product features change. This is the capstone of the Living Book initiative — the product literally keeps its own documentation current via the same workflow engine it documents.

When a feature ships (e.g., the workflow engine gets a new pattern), a scheduled or event-triggered workflow reads the updated codebase, regenerates the relevant chapter markdown, and creates a review task for human approval before publishing. This closes the "built by Stagent" loop: the app writes its own Book.

## User Story

As a Stagent maintainer, I want the Book to auto-update when features change, so documentation never falls out of date with the codebase.

As a reader, I want to trust that the code examples and patterns in the Book reflect the current version of Stagent, not a stale snapshot from months ago.

As a potential adopter, I want to see that Stagent's documentation was generated by Stagent itself, so the tool's capability is self-evident.

## Technical Approach

### Workflow Definition

A `book-chapter-update` workflow blueprint with this structure:

```
pattern: "planner-executor"
steps:
  1. [Planner] Detect which chapters need updating
     - Compare feature git history since last chapter generation
     - Map changed features to chapters via chapter-mapping.ts
  2. [Executor] For each stale chapter:
     a. Read current chapter markdown from docs/book/
     b. Read relevant source files (agent profiles, schemas, components)
     c. Generate updated chapter content preserving narrative voice
     d. Write updated markdown to docs/book/ch-{N}.md
     e. Update frontmatter lastGeneratedBy with task ID
  3. [Checkpoint] Create review task for human approval
     - Diff old vs new chapter content
     - Human approves, edits, or rejects
  4. [Finalizer] Update manifest.json timestamps
```

### Implementation Files

| File | Action | Purpose |
|------|--------|---------|
| `src/lib/book/update-detector.ts` | Create | Compares git log timestamps against chapter `lastGeneratedBy` to detect stale chapters |
| `src/lib/book/chapter-generator.ts` | Create | Agent prompt builder that assembles context (source files, existing chapter, strategy doc) for chapter regeneration |
| `src/lib/book/chapter-mapping.ts` | Modify | Add `sourceFiles` field mapping chapters to the codebase files they reference |
| Workflow blueprint JSON | Create | `book-chapter-update` workflow definition using planner-executor pattern |
| Schedule (optional) | Create | Weekly schedule to trigger the update workflow |

### Agent Profile

Use the `document-writer` agent profile for chapter generation:

- System prompt instructs the agent to maintain narrative voice, include real code examples, and follow the Problem → Solution → Implementation → Lessons structure from the strategy doc
- Context includes: current chapter markdown, relevant source files, `ai-native-book-strategy.md`, recent git commits for the feature area
- Output constraints: markdown format matching the parser conventions from `living-book-markdown-pipeline`

### Staleness Detection

```typescript
interface ChapterStaleness {
  chapterId: string;
  lastGenerated: string | null;     // from frontmatter
  latestSourceChange: string;       // from git log
  isStale: boolean;
  changedFiles: string[];           // source files that changed
}
```

### Human Review Gate

The workflow creates a review task (not auto-publishing):

- Task title: "Review updated Chapter {N}: {title}"
- Task description includes a diff summary of changes
- Attached document: the generated markdown file
- Approval publishes; rejection preserves the previous version

### UX Considerations

- **Book reader badge**: Chapters recently regenerated show a "Recently Updated" badge with the generation date
- **Transparency**: Each chapter's footer shows "Last generated by Stagent on {date}" with a link to the generating task
- **Manual trigger**: A "Regenerate Chapter" button in the Book reader (admin/power-user only) triggers the workflow for a single chapter

## Acceptance Criteria

- [ ] `update-detector.ts` correctly identifies stale chapters by comparing git timestamps to chapter metadata
- [ ] `chapter-generator.ts` produces markdown output matching the parser format from `living-book-markdown-pipeline`
- [ ] `book-chapter-update` workflow blueprint exists and can be executed via the workflow engine
- [ ] Workflow creates a human review task with diff summary before publishing
- [ ] Approved chapters are written to `docs/book/` with updated frontmatter
- [ ] Rejected chapters preserve the previous version unchanged
- [ ] "Recently Updated" badge shows on regenerated chapters in the Book reader
- [ ] Chapter footer displays generation metadata (date + task link)
- [ ] Manual "Regenerate Chapter" trigger available in Book reader
- [ ] Optional weekly schedule can be created to trigger automatic updates

## Scope Boundaries

**Included:**
- Staleness detection via git log comparison
- Chapter regeneration via document-writer agent
- Planner-executor workflow with human review checkpoint
- Generation metadata in frontmatter and UI
- Manual single-chapter regeneration trigger

**Excluded:**
- Real-time / webhook-triggered regeneration (batch only)
- Multi-language chapter generation
- A/B testing of generated vs. hand-written content
- Auto-generated screenshots or diagrams
- Cross-book linking to external documentation
