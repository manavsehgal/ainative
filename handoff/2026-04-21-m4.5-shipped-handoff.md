## Regression guards — don't undo these

**1. Scaffold-first ordering in `classifier.ts`.** The classifier checks scaffold triggers FIRST, then compose. Reversing this order makes ambiguous messages like *"build me a tool that pulls my GitHub issues"* classify as compose (wrong — user is asking for an integration). A future contributor "simplifying" the classifier by merging the two passes into one must preserve the scaffold-first branch.

**2. `inferScaffoldPlan` requires an integration noun.** The scaffold branch returns `null` (→ fall through to compose / conversation) when no `INTEGRATION_NOUNS` entry matches. A future contributor removing this guard makes every *"integrate with"* / *"connect to"* trigger fire the scaffold card regardless of whether a plausible plugin id can be built. `classifier.test.ts` `"connect to the idea"` case guards against this.

**3. `primitive-map.ts` entries must reference live registry ids.** The registry-validation test (`primitive-map.test.ts`) fails CI if a profile or blueprint is renamed without updating the map. A future contributor renaming `wealth-manager` → `portfolio-advisor` in the builtin registry must also update the map's `profileId` value, or CI breaks.

**4. `composition-hint.ts` must stay advisory.** Step 4 of the hint's actionable list is: *"If the user's actual need differs from this hint, prefer their stated intent over this hint."* Removing this line makes the hint prescriptive, which breaks the "LLM stays the dispatcher" contract. Composition-hint test `ends with the advisory fallback instruction` guards.

**5. Engine scaffold short-circuit has nested `try/finally` around `return`.** The engine's top-level `try/finally` (line ~836) unregisters the stream + cleans up — but we `return` early on scaffold before reaching it. The nested `try/finally` inside the scaffold branch ensures `unregisterChatStream` + `cleanupConversation` still fire. A future contributor removing the nested finally leaks the stream registration on every scaffold turn.

**6. `chat-message.tsx` renders `ExtensionFallbackCard` by reading `metadata.extensionFallback`, not by shared React context.** Mirrors the `composedApp` pattern (Phase 2+3). A future contributor consolidating the two renderers through a shared provider must preserve the metadata shape contract — the engine writes this, the component reads this, they're only coupled through the JSON blob.

**7. `chat-shell.tsx` listens for `ainative-chat-submit` CustomEvent.** This is the re-submit channel for the card's "Try this" button. A future contributor renaming the event to something shorter (`"chat-submit"`) without updating the card breaks the compose-alt path silently.

**8. `/api/plugins/scaffold` preserves Phase 6 error mapping.** The three named errors from `plugin-spec-tools.ts` (Invalid / AlreadyExists / Write) map to 400/409/500 with `code` fields. Future card UX can branch on `code` without re-parsing error messages. A contributor collapsing these to a generic `500 { error }` breaks future error-branch UX.

**9. Runtime-registry discipline.** No static import from `src/lib/chat/planner/*` or `src/app/api/plugins/scaffold/*` to anything reachable from `@/lib/agents/runtime/catalog.ts`. Verify with the `rg` grep in this handoff §"Automated smoke". Violating this joins the module-load-cycle hazard class documented in TDR-032.

**10. Settings toggle values.** `plugin-trust-model = auto | strict | off` (Phase 4 TDR-037). M4.5's planner doesn't consult this setting in v1; a future contributor wiring the planner to respect `strict` (e.g., "don't auto-suggest scaffolding in strict mode") must NOT collapse the three values into a boolean.

## Open decisions deferred to M4.6 / later

- **LLM-based intent classifier (v2)**: cheap Haiku pass on each inbound message before the pattern classifier. Pros: robust NLU, multi-turn, understands synonyms the curated phrase list misses. Cons: +1 LLM hop per turn (cost + latency), warrants a TDR, adds fallback handling. Defer until v1 accuracy ceiling is measured in real use.
- **Multi-turn composition understanding**: *"build me a reading list. Actually, make it also track ratings"* requires the classifier to see turn 2 as a refinement of turn 1. v1 treats each message atomically. M4.6 candidate.
- **Codex + Ollama engine parity**: v1 wires only the Claude SDK engine. Codex + Ollama users get unchanged behavior (they can still invoke composition via explicit tool calls, but not via NL). Straight lift for M4.6.
- **`plannerVerdict` telemetry**: add optional field to `recordTermination` entries for post-hoc accuracy tuning. v1 skipped — no DB/log surface added. M4.6 candidate once we have real chat traffic to analyze.
- **False-positive handling**: if pattern matching surfaces unacceptable false positives, add confidence scoring or a user-confirmation chip. Pre-ship heuristics expected OK; revisit if real chat traffic says otherwise.
- **Planner opt-out**: `AINATIVE_PLANNER_ENABLED=false` env kill switch. Not shipped in v1 — classifier is ~1ms + silent fall-through makes the cost of leaving it always-on negligible. M4.6 if needed.
